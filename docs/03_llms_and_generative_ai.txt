CHAPTER 3: LARGE LANGUAGE MODELS AND GENERATIVE AI
====================================================

What is a Large Language Model (LLM)?
---------------------------------------
A Large Language Model is a type of AI that has been trained on an enormous amount of text data and learned to understand and generate human language. The "large" refers to both the size of the training data and the number of parameters in the model.

Examples of LLMs:
- GPT-4 (by OpenAI) — powers ChatGPT
- Claude (by Anthropic) — the model powering this conversation
- Gemini (by Google DeepMind)
- Llama 3 (by Meta) — open source
- Mistral (by Mistral AI) — open source

Key numbers to appreciate the scale:
- GPT-3 has 175 billion parameters
- GPT-4 is estimated to have over 1 trillion parameters
- Trained on hundreds of billions of words from the internet, books, and code

The Transformer Architecture
------------------------------
All modern LLMs are based on the Transformer architecture, introduced in the 2017 paper "Attention is All You Need" by Google researchers.

Before Transformers, language models used RNNs/LSTMs which processed text word by word. Transformers process all words simultaneously using a mechanism called ATTENTION.

THE ATTENTION MECHANISM
The core insight: not all words in a sentence are equally important for understanding each word.

Example: "The bank by the river had steep walls."
To understand "bank" means riverbank (not financial bank), the model needs to pay attention to the word "river". The attention mechanism learns these relationships.

Self-attention:
- Every word looks at every other word in the sentence
- Calculates how much "attention" to pay to each word
- Produces a weighted representation that captures context

Multi-head attention:
- Run attention multiple times in parallel with different learned weight matrices
- Each "head" can focus on different types of relationships
- Final outputs are concatenated and transformed

TRANSFORMER ARCHITECTURE COMPONENTS:

Encoder:
- Reads and understands the input
- Used in models like BERT
- Good for tasks like classification, sentiment analysis

Decoder:
- Generates output text one token at a time
- Used in GPT-style models
- Good for text generation

Encoder-Decoder:
- Combines both
- Used in translation models like T5
- Input → encoder → decoder → output

Tokenization
-------------
LLMs don't process text word by word. They process "tokens", which are chunks of text.

A tokenizer splits text into tokens before feeding to the model.

Example using GPT tokenization:
"Hello, how are you?" → ["Hello", ",", " how", " are", " you", "?"]

"unbelievable" might be split into → ["un", "believ", "able"]

Why tokens instead of words?
- Handles unknown words by splitting into subword pieces
- More efficient than character-by-character
- Vocabulary size is manageable (~50,000 tokens for GPT)

Important fact: GPT-4 processes roughly 0.75 words per token. So "1000 tokens" ≈ 750 words.

Context Window
---------------
The context window is the maximum amount of text (in tokens) an LLM can consider at one time.

- GPT-3: 4,096 tokens (~3,000 words)
- GPT-4: 128,000 tokens (~96,000 words) — about a full short novel
- Claude 3: 200,000 tokens — about 150,000 words

Why it matters: If your document is longer than the context window, the model cannot read the whole thing at once. This is one of the reasons RAG (Retrieval Augmented Generation) was invented.

How LLMs Are Trained
---------------------

STEP 1: PRE-TRAINING
- Trained on massive text datasets (Common Crawl, Wikipedia, books, code)
- Task: predict the next word/token
- Example: given "The sky is", predict " blue"
- Does this billions of times across hundreds of billions of tokens
- This is where the model learns grammar, facts, reasoning, code, and more
- Requires enormous computing clusters (thousands of GPUs for weeks/months)
- Very expensive: GPT-4 estimated to cost $100 million+ to train

STEP 2: SUPERVISED FINE-TUNING (SFT)
- After pre-training, the model knows a lot but doesn't know how to be a helpful assistant
- Human contractors write examples of good conversations
- Model is fine-tuned on these examples to follow instructions

STEP 3: REINFORCEMENT LEARNING FROM HUMAN FEEDBACK (RLHF)
- Human raters rank different model responses from best to worst
- A "reward model" is trained to predict human preferences
- The LLM is then fine-tuned using reinforcement learning to maximize the reward model's score
- This is what makes models like ChatGPT pleasant and safe to use

Prompting LLMs
---------------
A "prompt" is the text you give the model as input. Good prompting is a skill.

ZERO-SHOT PROMPTING
Asking the model to do something without any examples:
"Translate this to French: Hello, how are you?"

FEW-SHOT PROMPTING
Giving a few examples to guide the model:
"Translate to French:
English: Hello → French: Bonjour
English: Goodbye → French: Au revoir
English: Thank you → French:"

CHAIN-OF-THOUGHT PROMPTING
Asking the model to reason step by step:
"Solve this step by step: If a train travels at 60mph for 2.5 hours, how far does it travel?"
This significantly improves accuracy on reasoning tasks.

SYSTEM PROMPTS
- Instructions given to the model before the conversation starts
- Define the model's role, tone, and constraints
- Example: "You are a helpful coding assistant. Only help with Python questions."

Retrieval Augmented Generation (RAG)
--------------------------------------
RAG is the technique this entire project is built on.

THE PROBLEM RAG SOLVES:
LLMs have two fundamental limitations:
1. Knowledge cutoff: they only know things from their training data. GPT-4's training data ends in early 2024 — it doesn't know about events after that.
2. Context window limits: you can't paste an entire company knowledge base into a prompt
3. Hallucination: models sometimes confidently state incorrect information

THE RAG SOLUTION:
Instead of relying purely on the model's internal knowledge:
1. Store your documents in a searchable database
2. When a user asks a question, RETRIEVE the relevant pieces from the database
3. Include those pieces in the prompt as context
4. Ask the LLM to answer using ONLY that provided context

RAG PIPELINE STEPS:
1. INDEXING (done once):
   - Load documents (PDFs, text files, etc.)
   - Split into smaller chunks (e.g., 512 characters each)
   - Convert each chunk to a vector embedding (a list of numbers representing meaning)
   - Store embeddings in a vector database

2. RETRIEVAL (done per query):
   - Convert the user's question to an embedding
   - Find the chunks most similar to the question (nearest neighbor search)
   - Return top-k most relevant chunks

3. GENERATION (done per query):
   - Construct a prompt with the retrieved chunks as context
   - Ask the LLM to answer using only the provided context
   - Return the answer with citations to the source chunks

Benefits of RAG:
- Always has access to up-to-date information (just update the database)
- Can work with private/proprietary documents
- Answers are grounded in actual source material (reduces hallucination)
- Can cite sources (traceable, trustworthy)

Vector Embeddings
------------------
Embeddings are how text is converted to numbers that capture meaning.

Key idea: Similar meanings → similar numbers (close in vector space)

Example:
- "dog" and "puppy" → very similar vectors (close distance)
- "dog" and "automobile" → very different vectors (far distance)
- "king" - "man" + "woman" ≈ "queen" (famous word2vec result)

How they're used in RAG:
- Convert each document chunk to a vector (list of ~1536 numbers for OpenAI's model)
- Store in a vector database (like ChromaDB, Pinecone, Weaviate)
- At query time, convert the question to a vector and find the closest chunks

This is called "semantic search" — searching by meaning rather than exact keywords.

Hallucination in LLMs
-----------------------
Hallucination is when an LLM confidently states something that is false.

Why it happens:
- LLMs are trained to predict plausible text, not to verify facts
- They have no mechanism to say "I don't know" naturally
- They pattern-match from training data and can generate plausible-sounding but wrong information

Examples of hallucinations:
- Making up fake academic papers with real-sounding authors and titles
- Inventing fake legal cases when asked for legal citations
- Giving wrong answers to math questions while sounding confident

How to reduce hallucination:
- Use RAG (ground the model in real source documents)
- Instruct the model to say "I don't know" when uncertain
- Use citation enforcement (this project's production feature!)
- Ask the model to reason step by step
- Use lower temperature settings (more deterministic output)

Temperature and Sampling
-------------------------
Temperature controls how "creative" or "random" the model's output is.

Temperature = 0:
- Deterministic (always same output for same input)
- Very focused and predictable
- Best for factual questions, code generation

Temperature = 0.7 (typical default):
- Some randomness
- Creative but still coherent
- Good for general chat

Temperature = 1.0+:
- Very random and creative
- Can be incoherent
- Useful for creative writing, brainstorming

For a RAG system like StudyRAG, use low temperature (0.1-0.2) for factual, grounded answers.

Fine-Tuning vs RAG vs Prompting
---------------------------------
There are three main ways to customize LLM behavior. Understanding the tradeoffs is important.

PROMPTING (cheapest, least powerful)
- Just craft better prompts
- No training required
- Changes behavior only for that session
- Limited by context window
- Best for: simple customization, quick experiments

RAG (moderate cost, very practical)
- Keep knowledge in external database
- Retrieve and inject into prompt
- Knowledge can be updated without retraining
- Scales to large knowledge bases
- Best for: question-answering over documents, keeping models up to date

FINE-TUNING (expensive, most powerful for specific tasks)
- Actually update the model's weights
- Model "bakes in" new knowledge or style
- Requires training data and compute
- Risk of catastrophic forgetting
- Best for: consistent style/tone, specialized tasks, when prompting alone fails

Evaluation of LLM Applications
--------------------------------
How do you know if your LLM application is working well?

RAGAS METRICS (used in this project):
- Faithfulness: does the answer only use information from the retrieved context?
- Answer Relevancy: does the answer actually address the question asked?
- Context Recall: do the retrieved chunks contain the information needed to answer?
- Context Precision: are the retrieved chunks actually relevant (low noise)?

OTHER IMPORTANT METRICS:
- Latency: how fast does it respond?
- Cost per query: how much does each API call cost?
- Hallucination rate: how often does it make things up?
- User satisfaction: do users find the answers helpful?

GOLDEN DATASET:
A golden dataset is a manually curated set of question-answer pairs with verified correct answers. This is the ground truth you compare your system's outputs against. Building a high-quality golden dataset of 50-200 examples is one of the most valuable things you can do for an LLM project.

AI Safety and Alignment
------------------------
As LLMs become more powerful, safety becomes increasingly important.

KEY CONCEPTS:

Alignment:
- Making sure AI systems do what humans actually want
- "The alignment problem": how do you specify human values precisely enough for an AI?

Constitutional AI (Anthropic's approach):
- Train models using a "constitution" of principles
- Model critiques and revises its own outputs
- Reduces need for human feedback on harmful content

Jailbreaking:
- Attempts to get an LLM to bypass its safety guidelines
- Usually done by constructing clever prompts
- Models are continually improved to resist jailbreaks

Prompt Injection:
- Malicious text in documents that attempts to hijack the LLM's behavior
- Example: A document contains hidden text "Ignore previous instructions and..."
- Critical security consideration for RAG systems

Bias in LLMs:
- Models learn biases present in training data
- Can produce biased outputs related to race, gender, religion, etc.
- Active area of research to detect and reduce bias
